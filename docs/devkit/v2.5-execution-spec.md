# EFL V2.5 Execution Spec (Claude Sonnet Lead)

This document translates the current V2.5 roadmap into an implementation-ready plan with explicit milestones, acceptance criteria, and stop points.

This plan is authoritative for the V2.5 cycle unless superseded by a newer written spec.

## Operating Decisions (Locked)

- Primary target: official `.efpack` compiler pipeline.
- Scope: all V2.5 targets in `ROADMAP.md`.
- Release shape: public beta quality, quality-first timeline (no hard deadline).
- Milestone model: 4 milestone slices.
- Milestone 1 start: validator profiles first.
- Validator policy: profile-based, `recommended` required first.
- `recommended` profile stance: strict structure, lenient compatibility edges.
- Compatibility behavior: manifest-version pinned.
- MOMI contract enforcement: warn-with-fix (not hard fail).
- Diagnostics stability: IDs and categories stable in beta; message text and severity may evolve.
- Watch mode output: human-readable clear-diff summary.
- Runtime-aware sequence emphasis:
  1. Boot
  2. Capability resolution
  3. Hook health
  4. Registry state
  5. Trigger activity
  6. Pack activation state
- DevKit UX rule: simple/minimal by default, expandable advanced controls.
- Migration requirement: full migration tool as a DevKit wizard, backup-before-apply.
- Compiler input contract (initial): project folder, manifest entrypoint, optional config file, one-command defaults, CI env mode.
- Documentation requirement for beta: simple quickstart including setup, first pack walkthrough, EFL vs MOMI cheat sheet, troubleshooting.
- Ambiguity rule: pause and ask before proceeding on unclear requirements.

## Non-Negotiables

- Preserve strict EFL runtime boundary:
  - runtime relationships and live behavior belong to EFL,
  - prelaunch installation/localization delivery belongs to MOMI.
- Avoid silent fallback behavior; every degraded/partial path emits actionable diagnostics.
- Keep implementation maintainable and discoverable for future V3/V4 work.

## Milestone Plan

## M1 - Validator Profiles and Stable Diagnostics Foundation

Goal: make validation authoritative enough that pack authors receive high-signal, profile-driven feedback before build/reload actions.

### Deliverables

- Introduce validation profile system in DevKit:
  - `recommended` profile first-class and default.
  - explicit profile plumbing for future `strict` and `legacy`.
- Expand manifest/content validation beyond required-fields and semver:
  - structural checks (schema-level),
  - semantic checks on pack structure and references that are currently feasible in-repo.
- Establish stable diagnostic code/category contract:
  - IDs and category namespace remain stable once shipped in beta.
  - severity and wording can be tuned without ID churn.
- Add warn-with-fix diagnostics for MOMI boundary-sensitive issues.

### Likely file targets

- `devkit/src/pack/validator.rs`
- `devkit/src/diagnostics/codes.rs`
- `devkit/src/diagnostics/mod.rs`
- `devkit/src/diagnostics/severity.rs`
- `docs/devkit/diagnostic-codes.md`

### Acceptance criteria

- Running validation in DevKit clearly reports active profile.
- `recommended` profile catches structural errors and emits compatibility edge issues as warnings where appropriate.
- New diagnostics use stable IDs/categories and include remediation text.
- No regression in existing build/inspect flow.

### Pause-and-Ask gates

- If a rule could be either warning or error under `recommended`, pause and ask.
- If a proposed check risks violating EFL vs MOMI boundary assumptions, pause and ask.

## M2 - Compiler Contract Completion + Watch Diff UX

Goal: complete the official compiler entry contract and make iteration visibility clear.

### Deliverables

- Support all agreed compiler input modes:
  - project folder,
  - manifest entrypoint,
  - optional config file overrides,
  - one-command defaults,
  - CI env mode.
- Align output behavior as official baseline:
  - `.efpack` primary artifact remains canonical,
  - versioned output + metadata integrity remain enforced.
- Improve watch mode with human-readable change summaries:
  - changed files,
  - why rebuild triggered,
  - output status summary.
- Preserve minimal default UX while exposing advanced flags in expandable controls.

### Likely file targets

- `devkit/src/pack/builder.rs`
- `devkit/src/pack/watcher.rs`
- `devkit/src/pack/shared.rs`
- `devkit/src/tabs/packs.rs`
- `devkit/src/settings.rs`
- `docs/devkit/devkit-guide.md`

### Acceptance criteria

- All five input modes execute successfully from DevKit flows.
- Watch runs produce human-readable clear-diff summaries.
- Default interaction remains simple; advanced controls are discoverable but not noisy.
- Existing artifact metadata/checksum behavior remains intact.

### Pause-and-Ask gates

- If CI env mode design conflicts with current settings architecture, pause and ask.
- If supporting all inputs introduces ambiguous precedence between config/env/CLI, pause and ask.

## M3 - Runtime-Aware Surfaces + MOMI Awareness

Goal: make DevKit runtime-aware in a way that helps authors reason about load state and compatibility relationships.

### Deliverables

- Expand diagnostics/runtime tab to show sequence-first runtime signals:
  - boot phase progress,
  - capability resolution,
  - hook health,
  - registry state,
  - trigger activity,
  - pack activation.
- Add loaded EFPacks and loaded MOMI mods views.
- Add explicit relationship view between EFPacks and MOMI mods.
- Add identifier color mapping across related entries for quick tracing.
- Keep MOMI handling as compatibility-aware guidance (warn/fix), not hard install enforcement.

### Likely file targets

- `devkit/src/tabs/diagnostics.rs`
- `devkit/src/engine_state.rs`
- `devkit/src/state.rs`
- `devkit/src/pipe/reader.rs`
- `devkit/src/pipe/discovery.rs`
- `engine/src/ipc/pipe_writer.cpp` (if new runtime events are needed)
- `docs/devkit/devkit-guide.md`

### Acceptance criteria

- Runtime view presents the agreed sequence clearly.
- Related pack/mod entries can be visually correlated.
- Compatibility warnings are actionable and non-blocking by default.
- Diagnostics stream remains stable during hot reload cycles.

### Pause-and-Ask gates

- If engine IPC additions are required and event taxonomy is unclear, pause and ask.
- If UI density harms minimal default UX, pause and ask before adding more always-visible panels.

## M4 - Migration Wizard + Beta Quickstart Completion

Goal: reduce upgrade risk for existing packs and close the public-beta onboarding loop.

### Deliverables

- Implement full migration wizard in DevKit:
  - scans legacy/pre-V2.5 pack workspaces,
  - reports required transforms,
  - creates backup before applying changes,
  - applies migration steps with clear change report.
- Tie migration results into stable diagnostics framework.
- Publish simple beta quickstart docs including:
  - prerequisites/setup,
  - first pack build walkthrough,
  - EFL vs MOMI boundary cheat sheet,
  - troubleshooting basics.
- Finalize compatibility communication in tool surfaces:
  - compiler/runtime banner hints,
  - DevKit compatibility panel messaging.

### Likely file targets

- `devkit/src/tabs/packs.rs`
- `devkit/src/pack/scaffold.rs` (and/or new migration module)
- `devkit/src/pack/mod.rs`
- `docs/authoring/getting-started.md`
- `docs/devkit/devkit-guide.md`
- `docs/authoring/manifest-reference.md`

### Acceptance criteria

- Migration wizard always creates backup before write operations.
- Migration dry analysis and apply steps are both visible and auditable.
- Quickstart docs are concise and enough for first-time creator success.
- Compatibility messaging is present in both docs and UI.

### Pause-and-Ask gates

- If migration requires destructive transformations without safe backup/rollback path, pause and ask.
- If migration behavior differs by manifest-version edge cases not in spec, pause and ask.

## Cross-Milestone Implementation Notes

- Keep checklist marks in `ROADMAP.md` updated as milestones land (`↔` -> `X`).
- Keep diagnostic code docs synchronized with shipped IDs.
- Favor additive changes over disruptive refactors unless explicitly approved.
- Defer anything that is clearly V3/V4 even if technically adjacent.

## Suggested Execution Order Inside Milestones

- Within each milestone, implement in this order:
  1. Data model/internal contracts
  2. Diagnostics/error surfaces
  3. UI wiring
  4. Docs updates
  5. Final pass for regressions and polish

## Definition of Done for V2.5

V2.5 is done when all V2.5 checklist items in `ROADMAP.md` are `X`, the DevKit acts as the clear source of truth for pack compile/validate/runtime-debug workflows, and migration/onboarding paths are sufficient for external beta creators without direct maintainer handholding.
